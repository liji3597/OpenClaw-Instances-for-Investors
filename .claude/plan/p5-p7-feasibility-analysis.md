# P5-P7 路线图可行性分析报告

## 执行摘要

**结论**: P5-P7 **技术上可行**，但不适合在当前"脚本 + SQLite + 模拟执行"架构上直接硬上主网。

**核心建议**: 必须先完成"执行内核重构"（订单状态机、签名服务、风控与审计分层），再分阶段推进；否则会在资金安全、重复执行、合规责任上出现 **Critical** 级风险。

---

## 1. 现状与范式切换分析

### 1.1 当前架构边界

```
Telegram/OpenClaw Gateway → 调脚本 → 写 SQLite → 外部依赖 Helius/CoinGecko/Jupiter
```

**当前设计特征**（见 README.md）：
- ✅ 非托管 + 只读 + 模拟执行
- ✅ 0 漏洞（已移除 bigint-buffer 依赖链）
- ✅ 系统授权门控（`--system` flag）
- ✅ 结构化错误处理（`MISSING_PARAMS` JSON 协议）

### 1.2 P5 范式切换本质

P5 引入真实签名与主网执行，属于**架构范式切换**而非功能补丁：

| 维度 | P0-P4 (模拟) | P5-P7 (真实上链) |
|------|-------------|-----------------|
| 信任模型 | 信息顾问 | 资金代理 |
| 错误成本 | 教育性 | 财务损失 |
| 数据流 | 意图→模拟→通知 | 意图→下单→风控→签名→广播→确认→对账→通知 |
| 状态管理 | pending/success/failed | 完整交易生命周期 |
| 合规要求 | 无 | KYC/AML/投资顾问牌照 |

## 3. 关键风险分析

### 3.1 Critical 🚨 级风险

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| **密钥管理失陷** | 云-based 私钥存储是 honeypot | 使用 Turnkey/Fireblocks/AWS Nitro Enclaves MPC |
| **重复下单** | 重试/并发导致错误成交 | 幂等键 + 状态机 + 至少一次交付 |
| **DeFi 协议风险** | 协议被 exploit 导致资金损失 | 协议保险 + 限额控制 + 白名单机制 |
| **监管归类** | 跟单/策略市场触发投资顾问牌照 | 法律审查 + 地理围栏 + 免责声明 |

### 3.2 Major ⚠️ 级风险

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| **SQLite 并发瓶颈** | 单机模型不适合真实交易并发 | 迁移至 PostgreSQL |
| **内存态指标丢失** | 重启即丢，不利于事故追溯 | 持久化至时序数据库 |
| **MEV/三明治攻击** | 高频交易可被 front-run | Jito MEV-protected bundles |
| **Slippage 失控** | 极端行情下成交价格偏离 | 硬编码滑点熔断 + 人工复核 |

## 4. 架构重构需求

### 4.1 当前架构限制

```
┌─────────────────────────────────────────┐
│  Telegram Bot                           │
└─────────────┬───────────────────────────┘
              │
┌─────────────▼───────────────────────────┐
│  OpenClaw Gateway                       │
│  • LLM 理解意图                          │
│  • 选择技能 → 执行脚本                    │
└─────────────┬───────────────────────────┘
              │
┌─────────────▼───────────────────────────┐
│  Skills (Node.js 脚本)                  │
│  • 直接操作 SQLite                       │
│  • 模拟执行记录                          │
└─────────────┬───────────────────────────┘
              │
┌─────────────▼───────────────────────────┐
│  SQLite (本地文件)                      │
└─────────────────────────────────────────┘
```

**问题**：
- 脚本直接操作数据库，无执行隔离
- 无统一风控切面
- 无审计日志链路
- 不适合并发交易

### 4.2 目标架构（推荐方案 B + C 混合）

```
┌─────────────────────────────────────────────────────────┐
│  Telegram Bot / Web Dashboard                           │
└─────────────┬───────────────────────────────────────────┘
              │
┌─────────────▼───────────────────────────────────────────┐
│  OpenClaw Gateway (编排层，保留)                          │
│  • 意图理解 + 技能路由                                     │
│  • 参数校验 + 用户确认                                     │
└─────────────┬───────────────────────────────────────────┘
              │ 意图 + 参数
┌─────────────▼───────────────────────────────────────────┐
│  ┌─────────────────────────────────────────────────┐    │
│  │  Execution Kernel (新增执行内核)                   │    │
│  │  • Order State Machine (订单状态机)               │    │
│  │  • Risk Control Layer (风控层)                   │    │
│  │  • Audit Logger (审计日志)                       │    │
│  └─────────────┬───────────────────────────────────┘    │
│                │ 签名请求                               │
│  ┌─────────────▼───────────────────────────────────┐    │
│  │  Signer Service (隔离签名服务)                    │    │
│  │  • MPC/HSM Provider (Turnkey/Fireblocks)        │    │
│  │  • Key Rotation (密钥轮换)                       │    │
│  └─────────────┬───────────────────────────────────┘    │
│                │ 签名后的交易                            │
│  ┌─────────────▼───────────────────────────────────┐    │
│  │  Transaction Lifecycle Manager                   │    │
│  │  • Broadcast → Confirm → Retry → Report         │    │
│  │  • Webhook Listener (Helius)                    │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
              │
┌─────────────▼───────────────────────────────────────────┐
│  PostgreSQL (替换 SQLite)                                │
│  • 订单表 + 审计表 + 风控表                                │
│  • JSONB 支持复杂 DeFi 头寸                                │
└─────────────────────────────────────────────────────────┘
```

### 4.3 核心新增组件

| 组件 | 职责 | 技术选型 |
|------|------|----------|
| Order State Machine | 管理订单生命周期 | 状态模式 + 事件驱动 |
| Risk Control Layer | 限额/熔断/预检 | 规则引擎 + 硬编码策略 |
| Signer Service | 隔离签名 + 密钥管理 | Turnkey/Fireblocks MPC |
| Audit Logger | 不可变审计日志 | Append-only PostgreSQL |
| Webhook Listener | 实时链上确认 | Helius Webhooks |

## 5. 实施路径与里程碑

### 5.1 三方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 增量改造 | 在现架构上硬上 | 上线快、改动小 | 安全与可审计性天花板低 |
| **B. 执行内核** (推荐) | 新增执行服务层 | 解耦、可扩展、支持 P5-P7 | 前期投入大 |
| C. 托管第三方 | 用托管签名/执行 | 缩短 P5 周期、合规快 | 供应商锁定、成本高 |

**推荐**: **B + C 混合** — 自建执行内核，签名先用托管 MPC，后续评估自建 HSM。

### 5.2 里程碑规划

```
M1 (2-4周) 架构奠基
├── 定义订单状态机 (CREATED → RISK_CHECK → SIGNING → BROADCAST → CONFIRMED)
├── 设计风控数据模型 (per-tx caps, daily limits, slippage thresholds)
├── 统一系统授权门禁 (所有写操作需 --system + 用户确认)
└── PostgreSQL 迁移计划

M2 (4-6周) P5 Alpha (白名单)
├── 接入 MPC 签名服务 (Turnkey/Fireblocks)
├── Jupiter Swap 真执行 (SOL/USDC 主网)
├── 余额预检 + Gas 估算
├── 失败重试机制 (exponential backoff)
└── 审计日志 v1 (tx signature, fill price, status)

M3 (2-4周) P5 Beta (低限额)
├── 日限额/滑点熔断硬编码
├── 人工复核流程 (高价值交易)
├── 回滚预案 + 紧急暂停按钮
├── 合规文档 (risk disclosure, ToS)
└── 10用户白名单测试

M4 (6-8周) P5 GA (全量)
├── 灰度发布 (whitelist → low-limit → general)
├── 监控告警体系完善
├── 事故响应流程
└── 保险/风险准备金

M5 (持续) P6 分层推进
├── 先做读侧能力 (检测/监控)
├── 低风险 DeFi (JitoSOL/mSOL 质押)
├── 再逐步开放 LP/借贷
└── 协议风险评估框架

M6 (持续) P7 机构化
├── Squads 多签集成 (Treasury 管理)
├── 合规报告导出 (CSV/PDF)
├── 策略市场 (低优先级，监管风险高)
└── 跟单交易 (低优先级，技术+监管复杂)
```

## 6. 技术债务与重构清单

### 6.1 数据库迁移

| 现状 | 目标 | 优先级 |
|------|------|--------|
| SQLite 文件 | PostgreSQL 服务 | Critical |
| 简单表结构 | 订单/审计/风控多表关联 | Critical |
| 无事务保证 | ACID + 外键约束 | Major |

### 6.2 架构解耦

| 现状 | 目标 | 优先级 |
|------|------|--------|
| 脚本直接操作 DB | 脚本 → Execution API | Critical |
| 无统一风控 | Risk Control Layer 切面 | Critical |
| 内存态指标 | 持久化时序数据 | Major |
| Polling 轮询 | Webhook 事件驱动 | Major |

### 6.3 安全加固

| 现状 | 目标 | 优先级 |
|------|------|--------|
| 不存储私钥 | MPC 托管签名 | Critical |
| 系统 flag 授权 | 多因子确认 (2FA) | Major |
| 基础日志 | 不可变审计日志 | Major |
| 无加密 | 传输 + 存储加密 | Major |

### 8.1 P5 完成标准

- [ ] 单笔交易 <$100 限额下，连续 30 天零事故

## 9. 结论与建议

### 9.1 核心结论

1. **技术上可行** — Solana 生态工具成熟 (Jupiter, Helius, Squads)
2. **架构需重构** — 不能直接硬上，需执行内核分层
3. **风险极高** — Critical 级风险集中在密钥管理、重试逻辑、DeFi 协议
4. **合规是瓶颈** — 策略市场/跟单可能触发监管

### 9.2 行动建议

| 优先级 | 行动 | 负责人 | 时间 |
|--------|------|--------|------|
| P0 | 完成 Execution Kernel 架构设计 | 架构师 | 1周 |
| P0 | PostgreSQL 迁移 | 后端 | 2周 |
| P0 | MPC 签名服务商选型 (Turnkey vs Fireblocks) | 安全 | 1周 |
| P1 | M1 里程碑实施 | 团队 | 4周 |
| P1 | 合规文档起草 + 法律审查 | 法务 | 2周 |
| P2 | M2 里程碑 (P5 Alpha) | 团队 | 6周 |

### 9.3 不推荐的功能 (建议砍掉或延后)

| 功能 | 原因 | 建议 |
|------|------|------|
| P7.2 跟单交易 | 实时性要求高 + 监管风险极高 | 砍掉或延至 12 个月后 |
| P7.1 策略市场 | 收益分成结算复杂 + 可能需牌照 | 延后至合规明确后 |
| P6.3 自动化收益耕作 | 无常损失 + 智能合约风险高 | 延后，先只做监控 |
